# scvalidate v0.4 — Rust plan for `recall`

Status: draft, 2026-04-19. Based on cProfile of `find_clusters_recall` on
epithelia 10k subsample (stratified on `subtype`), `max_iterations=6`,
`null_method=ZIP`.

## Baseline (profile on 8,511 cells × 16,337 genes)

| wall time | module |
|---:|---|
| 5903 s | recall total (w/ cProfile, ~30% profiler overhead) |
| 4461 s | recall total (prior 10k bench, no profiler) — v0.3 ship number |
| 117 s / 50 clusters | ROGUE 157k full (ship-ready, no Rust needed) |

Breakdown of the 5903 s run:

| share | hotspot |
|---:|---|
| **91.4%** | `_wilcoxon_per_gene` → `apply_along_axis(rankdata, 1, g_all)` |
| 89.0% | scipy `rankdata` (12.1M calls, single-row each) |
| ~27% | scipy `array_api_compat` dispatch overhead inside `rankdata` |
| ~14% | scipy `_contains_nan` (per-row NaN check — unnecessary) |
| 9% | numpy `argsort` |
| 3.2% | `_scanpy_preprocess_and_cluster` (6 iter: PCA + neighbors + Leiden) |
| 2.5% | `knockoff_threshold_offset1` (371 calls) |
| 1.8% | `sc.pp.neighbors` (part of the 3.2% above) |
| rest | knockoff sampling, logging, dict churn |

## Extrapolation to 157k

Rankdata is `O(n log n)` per row and the row count stays constant (~32.7k =
2 × n_genes). Cell count `n` grows 15.7×, so per-row `rankdata` cost grows
~18×. Pair count (~371) stays similar order.

- Py v0.3 @ 10k: 74 min
- Py v0.3 @ 157k extrapolated: **~22–25 h**

Need Rust.

## Target architecture — `scvalidate-rs`

One Rust crate exposed via PyO3. Scope limited to what dominates the profile:

### 1. `wilcoxon_ranksum_matrix` (primary target)

Signature (Python-side):

```python
def wilcoxon_ranksum_matrix(
    log_counts: np.ndarray,   # (2G, n_cells) f32, C-contiguous
    mask1: np.ndarray,        # (n_cells,) bool
    mask2: np.ndarray,        # (n_cells,) bool
) -> np.ndarray:              # (2G,) f64, two-sided p-values
```

Rust side:
- Slice to `g_all = log_counts[:, mask1 | mask2]` in caller (cheap, zero-copy
  via `np.ascontiguousarray` only when strictly needed)
- Parallelize over rows with `rayon::prelude::par_iter`
- Per row:
  - `argsort` (prefer `pdqsort` via `rayon::slice::ParallelSliceMut::par_sort_unstable_by`
    at the row level is OVERKILL; rows are short. Use `slice::sort_unstable_by` sequentially inside each row)
  - Tie-averaged ranks (scan sorted values, average rank over ties)
  - Sum ranks of `mask1`-cells → `U = R1 - n1*(n1+1)/2`
  - Normal-approx two-sided p-value (same formula as Python: `2 * (1 - Phi(|z|))`)
  - Clip `p ∈ [1e-300, 1.0]` exactly like the Python does
- Return `Vec<f64>` → numpy array via PyO3

Expected speedup on this function alone:
- Parallelism: 16 threads × ~80% efficiency = ~12×
- Native `argsort` + no scipy dispatch / no `_contains_nan` / no
  array_api_compat wrapping: ~5–8× on single-thread vs current scipy path
- Combined: **~60–100×**

### 2. `knockoff_threshold_offset1` (secondary)

150 s / 371 calls = 0.40 s each. Trivial Barber-Candès: sort `|w|`, scan to
find smallest threshold satisfying FDR. Rewriting in Rust gives maybe
10–20× → 8–15 s total. Not critical but tiny cost to implement once the
crate exists.

### 3. NOT in scope for v0.4

- `_scanpy_preprocess_and_cluster` (3% share — scanpy/Leiden is C already)
- `generate_knockoff_matrix` (not in top-40; ZIP sampling is fast)
- Full rewrite of `find_clusters_recall` control flow (keep Python driver)

## Projected v0.4 timings

Assume 60× on the 91% share, no change on the other 9%:

| metric | v0.3 10k | v0.4 10k target | v0.3 157k extrap | v0.4 157k target |
|---|---:|---:|---:|---:|
| wall time | 4461 s (74 min) | **~80 s** | ~22 h | **~22 min** |
| R parity (10k) | 649 s | (Py ~80s, **8× faster than R**) | — | — |

So v0.4 should beat R on recall at 10k (first time) and make 157k tractable
in one sitting.

## Crate layout

```
scvalidate_rust/          # PyO3 extension
├── Cargo.toml            # pyo3 + rayon + ndarray + numpy
├── src/
│   ├── lib.rs            # #[pymodule] fn scvalidate_rust(m): add funcs
│   ├── wilcoxon.rs       # rank-sum kernel
│   └── knockoff.rs       # threshold (phase 2)
└── pyproject.toml        # maturin build
```

Integration:
- `scvalidate/recall_py/core.py` imports `scvalidate_rust.wilcoxon_ranksum_matrix`
  (lazy) and falls back to the existing pure-Python path if import fails
- `tests/test_recall_parity.py` asserts p-values match scipy to within `1e-8`
  on a small toy matrix for both `mask1=mask2.flip` and ties-present cases
- `pyproject.toml` adds `scvalidate-rust ~= 0.1` as an optional dep;
  scvalidate keeps working without Rust, just slower

## Milestones

1. **M1 (Rust skeleton)** — PyO3 crate builds, empty module importable, CI
   wheel on Linux x86_64 + Windows + macOS via cibuildwheel. ~1 day.
2. **M2 (Wilcoxon kernel + parity)** — `wilcoxon_ranksum_matrix`
   implemented, tie-averaged, parallelized; parity test vs scipy passes
   (`|Δp| < 1e-8`). ~2 days.
3. **M3 (wire into recall)** — `_wilcoxon_per_gene` delegates to Rust when
   available; 10k bench re-run confirms ≥30× speedup. ~0.5 day.
4. **M4 (157k run + write-up)** — full 157k recall baseline completes
   in <30 min; per-cluster ROGUE/scSHC joined into the v0.4 `parity_v0.4.md`
   report. ~0.5 day.
5. **M5 (optional)** — Rust-port `knockoff_threshold_offset1`; GPU fallback
   via cupy (local only, behind feature flag). ~1 day.

## Open questions

- **Tie handling**: scipy's `rankdata` uses `average` by default. Rust port
  must match exactly — verify with ties-heavy toy (log1p of sparse counts
  produces many identical zeros → many ties).
- **Dtype**: scipy computes in f64. Rust kernel in f32 would halve memory
  but risks subtle p-value divergence at the 1e-300 floor boundary. Keep
  f64 by default, expose f32 as opt-in later.
- **Memory**: 2G × n = 32k × 157k × 8 B = 40 GB dense. For 157k the caller
  must already have this in RAM (recall does). Rust kernel is zero-copy via
  `numpy::PyReadonlyArray2`. OK on 56 GB WSL.
